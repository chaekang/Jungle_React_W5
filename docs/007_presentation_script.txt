[발표 대본]

안녕하세요. 저희 팀은 이번 정글 5주차 수요 코딩회 과제로 React의 핵심 기능인 Component, State, Hooks, 그리고 Virtual DOM의 diff와 patch를 직접 구현하고, 이를 활용한 Todo 데모 페이지를 만들었습니다.

이번 발표에서 말씀드릴 핵심은 세 가지입니다.
첫째, 저희는 React를 직접 사용하지 않고 핵심 개념을 Mini React로 구현했습니다.
둘째, 단순히 기능만 만드는 데서 끝나지 않고 내부 동작을 화면에서 설명할 수 있게 만들었습니다.
셋째, 실제 React와 비교했을 때 어떤 부분은 구현했고 어떤 부분은 의도적으로 단순화했는지 말씀드리겠습니다.

프로젝트 구조는 크게 두 부분입니다.
`src/mini-react`에는 직접 구현한 런타임 코어가 있고, `src/app`에는 그 코어를 사용하는 Todo 데모 앱이 있습니다.
코어 쪽에서는 `FunctionComponent` 클래스가 루트 렌더러 역할을 하며, `hooks[]` 배열을 저장하고 `mount()`와 `update()`를 담당합니다.

이번 구현에서 가장 중요했던 질문은, 함수형 컴포넌트는 매번 다시 실행되는데 state는 어떻게 유지되는가였습니다.
저희는 이 문제를 루트 `FunctionComponent`의 `hooks[]` 배열과 `hookIndex` 방식으로 해결했습니다.
렌더가 시작되면 현재 컴포넌트를 설정하고, `useState`, `useMemo`, `useEffect`가 순서대로 hooks 배열의 각 칸을 사용합니다.
그래서 함수는 다시 실행되더라도 state는 함수 안이 아니라 바깥의 hooks 배열에 남아 있게 됩니다.

과제 조건에 맞춰 state는 모두 루트 컴포넌트인 `App`에서만 관리했습니다.
예를 들어 `todos`, `inputText`, `editingId`, `filter` 같은 상태는 모두 App에 있고, `TodoInput`, `TodoList`, `TodoItem`, `TodoFooter` 같은 자식 컴포넌트는 props만 받아서 렌더링합니다.
즉, 자식은 state 없는 stateless component 구조로 설계했습니다.

이제 데모를 보겠습니다.
사용자는 Todo를 추가하고, 완료를 토글하고, 수정, 삭제, 필터 변경까지 할 수 있습니다.
여기까지는 일반 Todo 앱처럼 보이지만, 저희 팀의 차별점은 내부 동작을 같이 보여준다는 점입니다.

화면 오른쪽과 아래 패널을 보시면 현재 hooks 배열 상태를 코드처럼 볼 수 있고, 최근 diff와 patch 로그도 확인할 수 있습니다.
즉 버튼을 눌렀을 때 단순히 UI만 바뀌는 것이 아니라, 어떤 state가 바뀌고 어떤 DOM patch가 발생했는지까지 같이 추적할 수 있습니다.

예를 들어 Todo를 하나 토글해보겠습니다.
그러면 App의 state가 바뀌고, 새로운 Virtual DOM이 생성됩니다.
그다음 이전 Virtual DOM과 새 Virtual DOM을 비교해서 바뀐 부분만 patch하고, 실제 DOM에는 필요한 부분만 반영합니다.
이 과정을 로그로 보여주기 때문에 왜 다시 렌더링되었는지를 화면에서 바로 설명할 수 있습니다.

또 `useMemo`는 남은 Todo 개수와 필터링된 목록 계산에 사용했고, `useEffect`는 남은 Todo 수와 현재 필터가 바뀔 때 document title을 업데이트하도록 구현했습니다.
이 effect는 렌더 중 바로 실행되지 않고, DOM patch가 끝난 뒤 microtask 시점에 실행되도록 처리했습니다.

구현하면서 다른 프레임워크들의 diff 방식도 참고했습니다.
특히 key 기반 비교를 더 확장하거나, 컴포넌트 단위로 더 세밀하게 재사용하는 방향도 고민했습니다.
하지만 이번 과제는 state를 최상위 컴포넌트에서만 관리하고 자식은 props만 받는 구조를 전제로 했습니다.
그래서 자식 컴포넌트별 독립 state, 독립 Hook host, 인스턴스 재사용까지 포함한 고급 diff 전략은 그대로 반영하기 어려웠습니다.
대신 이번 프로젝트에서는 제약 안에서 hooks 배열, 상태 변화, diff와 patch의 핵심 원리를 명확히 설명할 수 있는 구조를 우선했습니다.

저희는 이 제약조건의 의도를, 기능을 줄이기 위한 제한이라기보다 상태 흐름을 단순하게 만들어 핵심 원리를 더 잘 보이게 하려는 장치라고 해석했습니다.
부모만 state를 가지면 데이터 흐름이 단순해지고, 자식은 props만 보게 되므로 상태 변화 추적이 쉬워집니다.
또 자식까지 각자 state를 가지게 하면 컴포넌트 인스턴스 보존, Hook host 분리, 렌더 순서 안정성까지 한 번에 다뤄야 해서 과제 범위가 급격히 커집니다.

마지막으로 실제 React와의 차이를 짧게 말씀드리겠습니다.
실제 React는 Fiber, batching, concurrent rendering 같은 훨씬 복잡한 스케줄링 구조를 가지고 있고, 자식 컴포넌트도 독립적인 상태를 가질 수 있습니다.
반면 저희 Mini React는 학습용 구현이기 때문에 상태 변경 시 즉시 `update()`를 호출하고, Hook도 현재는 루트에서만 안전하게 사용하도록 제한했습니다.
하지만 그 대신 Hook의 슬롯 재사용, state lifting, key 기반 diff, patch에 의한 최소 DOM 업데이트라는 핵심 개념은 직접 구현하고 설명할 수 있게 되었습니다.

그리고 구현에서 끝내지 않고 테스트도 함께 진행했습니다.
`vdom`, `diff`, `patch`, `hooks`, `component`, `app` 테스트를 작성했고, 총 27개 테스트를 통과했고 빌드도 성공했습니다.

정리하면, 저희 프로젝트는 React의 핵심 개념을 직접 구현한 Mini React 엔진과, 그 내부 동작까지 시각적으로 설명할 수 있는 Todo 데모 페이지입니다.
단순히 결과물만 만든 것이 아니라, 구현 원리와 검증 과정까지 함께 보여줄 수 있도록 준비했습니다.

이상 발표 마치겠습니다. 감사합니다.

[QnA 대비 짧은 답변 포인트]

Q. 왜 state를 App에만 두었나요?
A. 과제 조건이기도 했고, 상태를 한 곳에 모아야 데이터 흐름과 props 전달 구조를 가장 명확하게 설명할 수 있기 때문입니다.

Q. 왜 다른 프레임워크의 더 고급 diff 전략은 안 넣었나요?
A. 참고는 했지만, 이번 구조는 자식 컴포넌트가 독립 state를 가지지 않는 구조라 인스턴스 재사용까지 포함한 고급 전략을 그대로 넣기 어려웠습니다. 대신 과제의 핵심인 hooks, 상태 흐름, diff/patch 원리를 분명하게 보여주는 쪽을 우선했습니다.

Q. 실제 React와 가장 큰 차이는 무엇인가요?
A. 실제 React는 Fiber, batching, concurrent rendering 같은 고급 스케줄링이 있지만, 저희 구현은 학습용으로 핵심 개념만 단순화한 구조입니다.

Q. 이 프로젝트의 차별점은 무엇인가요?
A. Todo 기능 자체보다도 hooks 배열, diff/patch 로그, props 전달 구조를 화면에서 바로 볼 수 있게 만든 점이 가장 큰 차별점입니다.

Q. 가장 어려웠던 부분은 무엇인가요?
A. 함수가 다시 실행되어도 state를 유지해야 했기 때문에 hooks 배열의 인덱스를 안정적으로 관리하는 부분과, patch 이후 effect 실행 타이밍을 맞추는 부분이 가장 까다로웠습니다.
