[Mini React Todo Demo 발표 대본]

안녕하세요. 저희 팀은 이번 Week 5 과제로 React의 핵심 기능인 Component, State, Hooks, 그리고 Virtual DOM의 diff와 patch를 직접 구현한 뒤, 이를 활용해서 동작하는 Todo 데모 페이지를 만들었습니다.

이번 발표에서 보여드릴 핵심은 두 가지입니다.
첫째, 저희는 React를 사용하지 않고 핵심 개념을 직접 구현했습니다.
둘째, 단순히 기능이 동작하는 것에서 끝나지 않고, 내부 동작을 설명할 수 있는 데모 화면까지 만들었습니다.

먼저 구조부터 간단히 말씀드리겠습니다.
저희 프로젝트는 크게 두 부분으로 나뉩니다.
하나는 `src/mini-react` 폴더에 있는 라이브러리 코어이고, 다른 하나는 `src/app` 폴더에 있는 Todo 데모 앱입니다.

코어 영역에서는 `FunctionComponent` 클래스를 만들고, 이 안에서 `hooks` 배열을 관리합니다.
그리고 `mount()`는 첫 렌더링을, `update()`는 상태 변경 이후 재렌더링을 담당합니다.

핵심 질문은 "함수형 컴포넌트는 매번 다시 실행되는데 state는 어떻게 유지되는가?"였습니다.
저희는 이 문제를 hooks 배열과 hook index 방식으로 해결했습니다.
렌더링이 시작되면 현재 컴포넌트를 전역적으로 가리키고, `useState`, `useMemo`, `useEffect`가 순서대로 hooks 배열의 각 칸을 사용하도록 만들었습니다.
그래서 컴포넌트 함수는 다시 실행되더라도 state 값은 사라지지 않고 유지됩니다.

과제 조건에 맞춰 state는 모두 루트 컴포넌트인 `App`에서만 관리했습니다.
예를 들어 `todos`, `inputText`, `editingId`, `filter` 같은 값은 모두 App에 있고, `TodoInput`, `TodoList`, `TodoItem`, `TodoFooter` 같은 자식 컴포넌트는 props만 받아서 렌더링합니다.
즉, 자식은 stateless component로 설계했습니다.

이제 데모 화면을 보겠습니다.
첫 번째로 Todo를 추가할 수 있고, 완료 토글, 수정, 삭제도 가능합니다.
또한 전체, 진행 중, 완료 필터를 바꿀 수 있습니다.
이 부분은 사용자 상호작용에 따라 화면이 바뀌어야 한다는 과제 요구사항을 만족합니다.

하지만 저희 팀의 차별점은 여기서 끝나지 않습니다.
오른쪽과 아래 패널을 보면 현재 hooks 배열 상태를 코드처럼 확인할 수 있고, 최근 diff / patch 로그도 볼 수 있습니다.
즉, 버튼을 클릭했을 때 단순히 UI만 바뀌는 것이 아니라, 내부적으로 어떤 state가 바뀌고 어떤 patch가 발생했는지까지 추적할 수 있습니다.

예를 들어 Todo를 하나 토글해 보겠습니다.
그러면 App의 state가 바뀌고, 새로운 Virtual DOM이 만들어집니다.
그다음 이전 Virtual DOM과 새 Virtual DOM을 비교해서 바뀐 부분만 patch하고, 실제 DOM에는 필요한 부분만 반영합니다.
이 과정을 로그로 보여주기 때문에, "왜 다시 렌더링되었는가?"를 눈으로 확인할 수 있습니다.

또 하나 보여드릴 점은 `useMemo`와 `useEffect`입니다.
`useMemo`는 남은 Todo 개수와 필터링된 목록 계산에 사용해서 불필요한 재계산을 줄였습니다.
`useEffect`는 남은 Todo 개수와 현재 필터가 바뀔 때마다 브라우저 제목을 업데이트하도록 구현했습니다.
이 effect는 DOM patch가 끝난 뒤 microtask 시점에 실행되도록 처리했습니다.

테스트도 함께 준비했습니다.
`vdom`, `diff`, `patch`, `hooks`, `component`, `app`까지 각각 테스트를 작성해서 핵심 기능이 실제로 동작하는지 검증했습니다.
특히 hooks slot이 렌더링 이후에도 안정적으로 유지되는지, effect cleanup이 제대로 실행되는지, 수정 모드 버튼이 중복되지 않는지 같은 부분을 확인했습니다.

이번 프로젝트를 하면서 저희가 가장 크게 배운 점은 React를 사용할 때는 당연하게 느껴졌던 개념들이, 직접 구현해 보니 각각 분리된 설계 의도를 가지고 있었다는 점입니다.
예를 들어 왜 state를 부모에 올려야 하는지, 왜 Hook 호출 순서가 중요한지, 왜 key가 리스트 diff에서 필요한지를 코드 수준에서 이해할 수 있었습니다.

정리하면, 저희 프로젝트는 React의 핵심 개념을 직접 구현한 Mini React 엔진과, 그 내부 동작까지 시각적으로 설명할 수 있는 Todo 데모 페이지입니다.
단순히 결과물만 만든 것이 아니라, 구현 원리와 검증 과정까지 함께 보여줄 수 있도록 준비했습니다.

이상 발표 마치겠습니다. 감사합니다.

[QnA 대비 짧은 답변 포인트]

Q. 왜 state를 App에만 두었나요?
A. 과제 조건이기도 했고, 상태를 한 곳에 모아야 데이터 흐름과 props 전달 구조를 명확하게 설명할 수 있기 때문입니다.

Q. 실제 React와 가장 큰 차이는 무엇인가요?
A. 저희 구현은 학습용 Mini React라서 Fiber나 동시성 렌더링 같은 고급 기능은 없고, 핵심 개념을 단순화해 직접 구현한 버전입니다.

Q. 이 프로젝트의 차별점은 무엇인가요?
A. Todo 기능 자체보다도 hooks 배열, diff / patch 로그, props 전달 구조를 화면에서 바로 볼 수 있게 만든 점이 가장 큰 차별점입니다.

Q. 가장 어려웠던 부분은 무엇인가요?
A. 함수가 다시 실행되어도 state를 유지해야 했기 때문에, hooks 배열의 인덱스를 안정적으로 관리하는 부분과 업데이트 이후 effect 실행 타이밍을 맞추는 부분이 가장 까다로웠습니다.
