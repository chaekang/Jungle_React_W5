# CLAUDE.md

이 파일은 Claude Code가 이 프로젝트에서 작업할 때 따라야 할 규칙을 정의합니다.

---

## `conv/` 폴더 작성 규칙

### 목적

`conv/`는 **대화의 맥락과 의사결정의 배경**을 기록하는 폴더입니다.

- "왜 이 작업을 요청했는가"
- "어떤 배경에서 이 결정이 내려졌는가"
- "어떤 대안을 고려했고, 왜 이 선택을 했는가"

기술 명세(타입 정의, 알고리즘 설명, 코드 예시)는 `docs/`에 작성합니다.
`conv/`에는 그 결정에 이르게 된 **이유와 맥락**을 담습니다.

---

### 언제 작성하는가

- 사용자가 새로운 기능/구조/방향을 지시했을 때
- 여러 선택지 중 하나를 선택하는 의사결정이 발생했을 때
- 이전 결정을 번복하거나 변경했을 때
- 발표/데드라인/외부 제약 등 배경 맥락이 결정에 영향을 줬을 때

---

### 파일 명명 규칙

```
NNN_<주제>.md
```

예: `001_project_kickoff.md`, `002_jsx_format_decision.md`

번호는 대화 흐름 순서입니다. 하나의 대화 세션에서 여러 결정이 나오면 하나의 파일에 섹션으로 묶어도 됩니다.

---

### 파일 구조 템플릿

```markdown
# [주제]

## 배경

[사용자가 이 요청을 한 이유, 당시 상황, 해결하려는 문제]

## 요청 내용

[사용자가 요청한 것의 핵심]

## 고려한 선택지

| 선택지 | 장점 | 단점/이유 |
|---|---|---|
| A | ... | ... |
| B | ... | ... |

## 결정

[선택한 것과 그 이유. 기술적 이유 외에 발표, 일정, 팀 상황 등 맥락도 포함]

## 영향

[이 결정이 이후 작업에 미치는 영향]
```

### 좋은 conv 예시 vs 나쁜 conv 예시

**나쁜 예 (기술 명세 나열):**
```markdown
## JSX Pragma 결정

tsconfig.json에 jsxFactory: 'h'를 설정하면 <div>가 h('div', ...)로 변환된다.
각 파일에서 import { h } from '../mini-react'를 해야 한다.
```

→ 이건 `docs/`에 들어갈 내용입니다.

**좋은 예 (배경과 이유 중심):**
```markdown
## JSX Pragma 결정

## 배경

목요일 오전 발표에서 "우리가 React를 직접 만들었다"는 것을 직관적으로 보여줄 필요가 있었다.
h() 함수 직접 호출 방식으로도 동작하지만, 실제 React와의 유사성을 코드 수준에서 보여주기 어렵다.

## 결정

JSX pragma 방식 선택.
`<div>` → `h('div', ...)` 변환 원리를 tsconfig 한 줄로 증명할 수 있고,
발표 시 실제 React 코드와 우리 코드를 나란히 놓아도 자연스럽다.

## 영향

모든 컴포넌트 파일이 .tsx 확장자를 사용해야 하고,
h를 반드시 import해야 한다는 불편함이 생기지만, 발표 가치가 더 높다고 판단.
```

---

### 기억할 것

- `conv/`는 나중에 "왜 이렇게 됐지?"라고 물었을 때 답이 되는 파일입니다.
- 기술 스펙이 아닌 **의사결정 로그**입니다.
- 짧아도 됩니다. 배경과 이유가 명확하면 충분합니다.
- 대화가 끝날 때 주요 결정이 있었다면 반드시 기록합니다.
