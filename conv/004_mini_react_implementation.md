# Mini React 구현 구조 결정

## 배경

- `docs/002_PRD.md` 기준으로 Week 5 과제를 바로 구현해야 했다.
- 현재 저장소에는 명세 문서만 있고, 실행 가능한 앱/테스트/빌드 설정은 없는 상태였다.
- 발표 전까지 빠르게 완성하면서도 각 핵심 모듈을 설명 가능한 구조가 필요했다.

## 요청 내용

- PRD를 읽고 적절한 하위 계획을 세운 뒤, 그 순서대로 구현한다.

## 고려한 선택지

- PRD 순서를 그대로 9단계로 잘게 나눠 순차 구현
- 초기 세팅, 코어 라이브러리, Todo 앱으로 묶어 큰 단위로 진행
- 앱부터 먼저 만들고 필요한 런타임 기능을 뒤에서 채우는 방식

## 결정

- 작업 단위는 3단계로 묶었다.
- 1단계: `Vite + TypeScript + Vitest` 기반 세팅
- 2단계: `mini-react` 코어(`vdom`, `diff`, `patch`, `hooks`, `component`)와 테스트
- 3단계: Todo 앱 UI와 브라우저 연결
- 이유:
  - 저장소가 비어 있어 세팅 없이는 어떤 모듈도 검증할 수 없었다.
  - 코어를 테스트로 먼저 고정해야 Todo 앱에서 생기는 문제를 앱 로직과 런타임 로직으로 분리할 수 있었다.
  - PRD의 학습 포인트인 `State`, `Hooks`, `Virtual DOM + Diff + Patch`를 모듈 단위로 설명하기 쉬운 구조가 됐다.

## 영향

- `tests/` 기준으로 핵심 동작을 먼저 검증한 뒤 앱을 얹는 흐름이 만들어졌다.
- 앱은 루트 `App`만 상태를 가지도록 설계해 PRD의 `Lifting State Up` 제약을 지켰다.
- JSX pragma를 유지하면서도 함수형 자식 컴포넌트를 바로 호출하는 방식으로 단순한 커스텀 렌더러를 구현했다.
